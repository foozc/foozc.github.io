| **grilling** | 任务开始前的盘问式需求澄清，确保对齐再动手 |
| **tracer-bullet** | 最小可行路径的探索性实现，先打通一条完整链路 |

## 架构原则（按优先级排列）

1. **SRP（单一职责）**：每个文件只做一件事。数据文件只管数据，组件只管渲染
2. **OCP（开闭原则）**：新增功能不修改已有代码，通过扩展新模块实现
3. **DIP（依赖倒置）**：组件依赖数据抽象（`import`），不依赖具体实现
4. **DRY（不要重复）**：CSS 变量集中定义，避免硬编码值
5. **KISS（保持简单）**：优先原生 Vue 特性，不引入不必要的依赖

## 文件结构地图

```
src/
  App.vue          根组件，状态管理 + 条件路由
  components/      页面组件（OCP：新增不修改旧组件）
  data/            数据模块（SRP：每个文件一个职责）
  games/           游戏组件
index.html         HTML 入口
```

## 技术栈速查

- Vue 3 (Composition API + `<script setup>`)
- Vite 5 (ESBuild + HMR)
- 无 Vue Router（用 `v-if` 条件渲染替代）
- 无状态管理库（用 Vue `ref` 管理全局状态）
- CSS 变量驱动亮色/暗色主题

## 沟通协议

### 1. 任务开始前：Grilling（盘问式澄清）

当我收到新任务时，我会主动提问以缩小模糊地带。例如：

> **理解确认**："我理解您想修改 X，具体是 A 方案还是 B 方案？"
> **边界确认**："这个变更影响范围是只改 Y 文件，还是会波及 Z？"
> **验收标准**："怎么算完成？有没有明确的检查清单？"

### 2. 执行过程中：小步反馈循环

- **一次只改一个文件/一个模块**，每步都等确认
- **构建验证**：每次修改后 `npm run build` 确保不坏
- **变更透明**：每次提交前说明改了什么、为什么改、影响范围

### 3. 任务完成后：结果确认

- **提供明确的结果摘要**：改了什么文件、什么内容
- **提供验证方式**：如何确认改对了（构建、预览、部署）
- **保留回溯能力**：每步改动都有 git commit

### 4. 共享语言优先

- 使用本项目术语表中的词汇（如 "resume"、"architecture"、"ADR"）
- 对于不明概念，先问清术语定义再继续
